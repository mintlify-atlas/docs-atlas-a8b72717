The Grok Search MCP Server includes comprehensive error handling with automatic retry logic, detailed error messages, and graceful degradation.

## Error response structure

All errors return a standardized JSON format with diagnostic information:

```json
{
  "error": "Detailed error message",
  "status": "failed",
  "query": "original query",
  "search_type": "web",
  "analysis_mode": "basic",
  "from_date": null,
  "to_date": null,
  "timestamp": "2025-06-24T12:00:00.000Z",
  "request_id": "req_1234567890_abc123"
}
```

<Note>
The `request_id` is unique for each request and useful for debugging and support inquiries.
</Note>

## Common error types

### API key missing

**Error message**: `"API service is not healthy - missing XAI_API_KEY"`

**Cause**: The `XAI_API_KEY` environment variable is not set or is empty.

**Resolution**:

1. Verify your API key is set in the Claude Desktop configuration:

```json
{
  "mcpServers": {
    "grok-search": {
      "command": "npx",
      "args": ["grok-search-mcp"],
      "env": {
        "XAI_API_KEY": "xai-your-actual-key-here"
      }
    }
  }
}
```

2. Restart Claude Desktop completely
3. Use `health_check` tool to verify API connectivity

<Warning>
The server will not process any requests without a valid API key. All tool calls will fail immediately with this error.
</Warning>

### Validation failures

**Error message**: Various validation error messages

**Common examples**:
- `"Search query must be a non-empty string"`
- `"Search query too long (max 1000 characters)"`
- `"from_date must be in ISO8601 format (YYYY-MM-DD)"`
- `"from_date must be before or equal to to_date"`

**Resolution**: Fix the input parameters according to the error message.

<Accordion title="Query validation">
```javascript
// From index.js lines 196-207
if (!query || typeof query !== 'string') {
  throw new Error("Search query must be a non-empty string");
}

const sanitizedQuery = query.trim().replace(/[\x00-\x08\x0B\x0C\x0E-\x1F\x7F]/g, '');
if (sanitizedQuery.length === 0) {
  throw new Error("Search query cannot be empty after sanitization");
}

if (sanitizedQuery.length > 1000) {
  throw new Error("Search query too long (max 1000 characters)");
}
```

**Checks**:
- Query must be a string
- Query cannot be empty after trimming
- Query cannot exceed 1000 characters
- Control characters are removed
</Accordion>

### Rate limits

**Error message**: `"API request failed: 429 - Rate limit exceeded"`

**Cause**: Too many requests to the xAI API in a short period.

**Automatic handling**: The server automatically retries with exponential backoff:

```javascript
// From index.js lines 152-162
if ((response.status >= 500 || response.status === 429) && retryCount < this.maxRetries) {
  const backoffDelay = Math.min(1000 * Math.pow(2, retryCount), 10000);
  Logger.warn(`Request failed, retrying in ${backoffDelay}ms`, { 
    status: response.status, 
    attempt: retryCount + 1,
    maxRetries: this.maxRetries 
  });
  
  await new Promise(resolve => setTimeout(resolve, backoffDelay));
  return this.makeRequest(endpoint, data, retryCount + 1);
}
```

**Resolution**: The server handles this automatically. Wait for the retry logic to complete.

<Tip>
**Retry delays**:
- 1st retry: 1 second
- 2nd retry: 2 seconds
- 3rd retry: 4 seconds
- Maximum delay: 10 seconds
</Tip>

### Timeouts

**Error message**: `"Request timeout after 30000ms"`

**Cause**: The API request took longer than the configured timeout (default: 30 seconds).

**Resolution**:

1. Increase the timeout in your configuration:

```json
{
  "mcpServers": {
    "grok-search": {
      "env": {
        "XAI_API_KEY": "your-key",
        "GROK_TIMEOUT": "45000"  // 45 seconds
      }
    }
  }
}
```

2. Use basic mode instead of comprehensive mode (faster)
3. Reduce `max_results` parameter
4. Check your internet connection

<Warning>
Timeout errors trigger automatic retries but may still fail if network conditions persist.
</Warning>

### Network errors

**Error message**: `"Failed to make API request: [network error details]"`

**Common causes**:
- No internet connection
- DNS resolution failure
- Firewall blocking requests
- xAI API service outage

**Automatic handling**: The server retries network errors up to 3 times:

```javascript
// From index.js lines 177-187
if (retryCount < this.maxRetries && !error.message.includes('API request failed:')) {
  const backoffDelay = Math.min(1000 * Math.pow(2, retryCount), 10000);
  Logger.warn(`Network error, retrying in ${backoffDelay}ms`, { 
    error: error.message, 
    attempt: retryCount + 1 
  });
  
  await new Promise(resolve => setTimeout(resolve, backoffDelay));
  return this.makeRequest(endpoint, data, retryCount + 1);
}
```

**Resolution**:
1. Verify internet connectivity
2. Check if xAI API is operational
3. Wait for automatic retries to complete
4. Try again after a few minutes

## Automatic retry logic

The server implements exponential backoff for transient failures (implemented in `makeRequest` function, lines 122-192).

### Retry conditions

Automatic retries occur for:
- **5xx server errors**: API service temporarily unavailable
- **429 rate limit errors**: Too many requests
- **Network errors**: Connection failures, timeouts

### Retry configuration

**Default settings**:
- Maximum retries: 3 attempts
- Initial delay: 1 second
- Backoff multiplier: 2x per retry
- Maximum delay: 10 seconds

**Custom configuration**:

```json
{
  "env": {
    "GROK_MAX_RETRIES": "5"  // Increase max retries to 5
  }
}
```

### Retry flow diagram

<Tabs>
  <Tab title="Successful retry">
    1. Initial request fails (429 rate limit)
    2. Wait 1 second
    3. Retry #1 fails (429 rate limit)
    4. Wait 2 seconds
    5. Retry #2 succeeds ✓
    6. Return results
  </Tab>
  <Tab title="All retries exhausted">
    1. Initial request fails (500 server error)
    2. Wait 1 second
    3. Retry #1 fails (500 server error)
    4. Wait 2 seconds
    5. Retry #2 fails (500 server error)
    6. Wait 4 seconds
    7. Retry #3 fails (500 server error)
    8. Return error to user ✗
  </Tab>
</Tabs>

<Note>
Retry logic does NOT apply to validation errors or authentication failures. These fail immediately.
</Note>

## Handling errors in your application

### Check for error responses

```javascript
const response = await callTool('grok_search', {
  query: 'AI developments',
  analysis_mode: 'comprehensive'
});

const result = JSON.parse(response.content[0].text);

if (result.error || result.status === 'failed') {
  console.error(`Search failed: ${result.error}`);
  console.error(`Request ID: ${result.request_id}`);
  // Handle error appropriately
  return;
}

// Process successful results
const analysis = result.comprehensive_analysis;
```

### Categorize errors

```javascript
function categorizeError(errorMessage) {
  if (errorMessage.includes('API key')) return 'API_KEY';
  if (errorMessage.includes('rate limit')) return 'RATE_LIMIT';
  if (errorMessage.includes('timeout')) return 'TIMEOUT';
  if (errorMessage.includes('format') || errorMessage.includes('validation')) {
    return 'VALIDATION';
  }
  return 'UNKNOWN';
}

const errorType = categorizeError(result.error);

switch (errorType) {
  case 'API_KEY':
    // Guide user to check configuration
    break;
  case 'RATE_LIMIT':
    // Wait and retry
    break;
  case 'TIMEOUT':
    // Suggest using basic mode
    break;
  case 'VALIDATION':
    // Fix input parameters
    break;
}
```

### Graceful degradation

```javascript
async function searchWithFallback(query) {
  try {
    // Try comprehensive mode first
    const result = await search(query, 'comprehensive');
    return result;
  } catch (error) {
    if (error.message.includes('timeout')) {
      // Fallback to basic mode on timeout
      console.log('Falling back to basic mode...');
      return await search(query, 'basic');
    }
    throw error;
  }
}
```

```json
  "success_rate": "98.00%",

    "cacheSize": 12,
```

**Unhealthy response**:

</Tip>

## Best practices

### Always check for errors

Never assume requests succeed. Always check the response structure:

```javascript
if (result.error) {
  // Handle error case
} else {
  // Process results
}
```

### Use appropriate timeouts

- **Basic mode**: 30 seconds (default) is usually sufficient
- **Comprehensive mode**: Consider 45-60 seconds for complex queries
- **High-latency networks**: Increase timeout accordingly

### Log error details

Capture `request_id` and full error message for debugging:

```javascript
if (result.error) {
  console.error({
    error: result.error,
    request_id: result.request_id,
    timestamp: result.timestamp,
    query: result.query
  });
}
```

### Implement user feedback

Provide clear error messages to users:

  </Tab>
  <Tab title="Bad">
    "Error: Request timeout after 30000ms"

</Tabs>
